How to Collaborate on a GitHub Repository

A practical guide to roles, branching, issues, pull requests, reviews, and conflict resolution.

1) Choose a Collaboration Model

Collaborators (same repo)

  • Owner adds you as a Collaborator (Repo → Settings → Collaborators).
  • You push branches to the main repo: origin.
  • Best for private or team repos.

Fork & Pull (open source)

  • You fork the repo to your account.
  • Clone your fork, open a PR from your fork back to the original.
  • Best for public/open-source contributions.

2) Recommended Branching Workflow

  1. Sync main 
    git switch main
git pull origin main
  2. Create a feature branch 
    git switch -c feature/login-ui
  3. Commit small, focused changes 
    git add .
git commit -m "feat(login): add login form and validation"
  4. Push your branch 
    git push -u origin feature/login-ui
  5. Open a Pull Request (PR) on GitHub: compare feature/login-ui → main.
Tip: Use prefixes like feat/, fix/, chore/, and docs/ for clarity.

3) Issues: Plan & Track Work

  • Create an Issue with a clear title, steps to reproduce (if a bug), acceptance criteria.
  • Use labels like bug, feature, good first issue.
  • Assign yourself and link your PR to the Issue (e.g., write “Closes #123” in PR description).

4) Pull Requests (PRs): Propose Changes

  • Keep PRs small and focused; add a meaningful description and screenshots if UI changes.
  • Request reviewers; address comments with follow-up commits.
  • Ensure checks pass (lint/tests/CI) before merge.
# Create PR from CLI (GitHub CLI):
gh pr create --base main --head feature/login-ui --title "Login UI" --body "Closes #123"
Tip: Add a PULL_REQUEST_TEMPLATE.md to standardize PR quality.

5) Code Review Etiquette

  • Authors: be receptive, respond to each comment, and document decisions.
  • Reviewers: be constructive, suggest improvements, approve or request changes.
  • Use suggestions in reviews so authors can apply with one click.

6) Keep Your Branch Up to Date

Merge main into your branch

git switch feature/login-ui
git fetch origin
git merge origin/main

Or rebase onto main (linear history)

git fetch origin
git rebase origin/main
# Resolve conflicts, then:
git push --force-with-lease

Use merge for shared branches; rebase for cleaner history. Never rebase public branches others are using.

7) Resolve Merge Conflicts

  1. Run your merge/rebase; Git will pause on conflicts.
  2. Open conflicted files and edit sections between <<<<<<< and >>>>>>>.
  3. Stage and continue: 
    # For merge:
git add .
git commit

# For rebase:
git add .
git rebase --continue
Heads-up: After a rebase, push with --force-with-lease (safer) to update the remote branch.

8) Sync a Fork (if using Fork & PR)

# Add the original repo as upstream
git remote add upstream https://github.com/ORIGINAL_OWNER/REPO.git

# Fetch and update your main
git fetch upstream
git switch main
git merge upstream/main   # or: git rebase upstream/main

# Push your updated main to your fork
git push origin main

9) Protect the Main Branch (Owner/Maintainers)

  • Enable Branch protection (Repo → Settings → Branches).
  • Require PR reviews and passing status checks before merge.
  • Disallow force-push and deletion of main.

11) Communication & Hygiene

  • Use Issues and PR comments, mention teammates with @username.
  • Write clear commit messages (e.g., Conventional Commits).
  • One topic per branch/PR; keep changes reviewable (< ~300 lines when possible).

Quick commands recap: 

# Start feature work
git switch main && git pull
git switch -c feature/calc

# Push & open PR
git push -u origin feature/calc

# Update your branch
git fetch origin
git merge origin/main   # or: git rebase origin/main

# Resolve conflicts and continue
git add . && git commit        # merge flow
# or
git rebase --continue          # rebase flow
