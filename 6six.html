What is git revert?

The git revert command creates a new commit that undoes the changes introduced by a previous commit. Unlike git reset, it does not delete history — instead, it adds a safe "undo" commit.

Why use it?

  • To safely undo changes in a shared/public branch.
  • To preserve commit history for better tracking.
  • To avoid rewriting history (important when others have pulled your branch).

Basic Usage

1. Revert a specific commit

git revert <commit-hash>

This will open the default editor to write a commit message for the revert.

2. Revert without opening an editor

git revert <commit-hash> --no-edit

3. Revert multiple commits

git revert <oldest-commit>^..<newest-commit>

4. Revert the latest commit

git revert HEAD

Example Workflow

# View commit history
git log --oneline

# Suppose the commit hash is abc123
git revert abc123

# Git will create a new commit that undoes abc123

Important Notes

  • Revert is non-destructive — it doesn’t remove commits.
  • Good for undoing changes in branches used by others.
  • If there are conflicts during revert, Git will ask you to resolve them before committing.
  • For local private branches, git reset might be faster if you don't care about keeping history.

Difference Between Revert and Reset

git revert   # Creates a new commit that undoes previous changes
git reset    # Moves HEAD to a previous commit, removing commits from history (dangerous in shared repos)
