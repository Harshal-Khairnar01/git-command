What is git reset?

First change for reset

The git reset command moves the current branch's HEAD to a specified commit. It can modify the staging area, the working directory, or both — depending on the mode used.

Why use it?

  • To undo commits or changes locally.
  • To remove unwanted commits before pushing.
  • To unstage files without losing changes.

Modes of git reset

1. --soft

git reset --soft <commit-hash>

Moves HEAD to the given commit, keeps changes staged (in index).

2. --mixed (default)

git reset --mixed <commit-hash>
# or simply:
git reset <commit-hash>

Moves HEAD to the given commit, unstages changes but keeps them in working directory.

3. --hard

git reset --hard <commit-hash>

Moves HEAD to the given commit, discards all staged and working directory changes.

Examples

# View commit history
git log --oneline

# Go back to previous commit but keep changes staged
git reset --soft HEAD~1

# Go back and unstage changes
git reset HEAD~1

# Go back and discard all changes
git reset --hard HEAD~1

Important Notes

  • --hard will permanently delete changes not committed — use carefully.
  • git reset changes commit history; avoid using it on shared/public branches.
  • Use git reflog to recover commits if you reset by mistake.

Difference Between reset, revert, and restore

git reset   # Move HEAD to an earlier commit, can modify history (dangerous in shared repos)
git revert  # Create a new commit that undoes an earlier commit (safe in shared repos)
git restore # Restore file contents from commit or discard changes
